# 105: Задача zero_shard_states (track: distributed, Medium)

## Зависит от
000-conventions, 010, 030, 101, 102 (использует байт-арифметику)

## Что сделать
`torch_judge/tasks/zero_shard_states.py` +
`templates/54_zero_shard_states.ipynb` + solution.

## Поведение
```python
def zero_shard_plan(
    named_param_numels: list[tuple[str, int]],  # (имя, numel), порядок фикс.
    world_size: int,
    stage: int,                                  # 1 | 2 | 3
) -> dict  # план шардирования + память
```
Жадное распределение ПАРАМЕТРОВ по ранкам: очередной параметр — ранку с
минимальной текущей суммой numel (при равенстве — меньший rank).
Возвращаемая структура:
```python
{
  "assignment": {rank: [param_names...]},
  "bytes_per_rank": {rank: int},   # см. ниже
  "replicated_bytes_per_param": int,  # что НЕ шардится на этом стейдже
}
```
Память на ранк (bf16 mixed AdamW с master weights, как в 102):
- stage 1: шардятся optimizer states + master (4+4+4=12 байт/парам —
  только для своих параметров); params (2) и grads (2) реплицированы
  на все ранки.
- stage 2: шардятся optimizer states + master + grads (12+2); params (2)
  реплицированы.
- stage 3: шардится всё (16 байт/парам только для своих).
`bytes_per_rank[r]` = шардированные байты своих параметров +
реплицированные байты ВСЕХ параметров. Невалидный stage → ValueError.

## Тесты judge (минимум)
1. Консервация: sum шардированной части по ранкам == шардированные байты
   всех параметров (для каждого стейджа).
2. Баланс: max-min нагрузки по numel ≤ max(numel) параметров (свойство
   жадного алгоритма).
3. Монотонность стейджей: bytes_per_rank(stage3) ≤ stage2 ≤ stage1 для
   world_size ≥ 2 (на фикстуре из 6 разных параметров).
4. world_size=1: любой stage даёт полные 16 байт/парам × все параметры.
5. Детерминизм: повторный вызов — идентичный план; stage=4 → ValueError.

## Подсказки (3 уровня)
1. «Выпиши для каждого стейджа: что лежит на каждом ранке, что — только
   на своём».
2. Таблица стейдж × компонент (P/G/OS+master) × шардится?
3. Жадное «в наименее загруженный ранк» — это partition problem; кода
   не давать, только идею кучи/минимума.

## Приёмка
Стандартная (000) + solution-ноутбук содержит проверочную ячейку,
воспроизводящую известное соотношение из ZeRO-статьи (память stage3 ≈
1/N от baseline при равных параметрах) на игрушечном примере.

## Не делать
Не моделировать коммуникацию (это в теории 101 и вопросах); не шардить
внутри параметра (гранулярность — целый параметр, у настоящих ZeRO/FSDP
иначе — сказать об этом в условии одной строкой).
