## Поведение
```python
def ring_allreduce(
    rank_tensors: list[torch.Tensor],   # длина N, одинаковые shape
    on_send: Callable[[int, int], None] | None = None,  # (src, dst)
) -> list[torch.Tensor]
```
Симуляция ring all-reduce БЕЗ torch.distributed, в одном процессе:
1. Каждый «ранк» делит свой тензор на N чанков (последняя часть может
   быть короче — использовать `torch.tensor_split`).
2. Фаза reduce-scatter: N-1 итераций, ранк i отправляет чанк соседу
   (i+1)%N; после фазы у каждого ранка один полностью
   просуммированный чанк.
3. Фаза all-gather: N-1 итераций передачи готовых чанков по кольцу.
4. Итог: каждый выходной тензор равен СРЕДНЕМУ входных (сумма / N).
`on_send(src, dst)` вызывается на каждую логическую пересылку чанка.
Входные тензоры не мутировать (работать на копиях).

## Тесты judge (минимум)
1. Корректность: N=4, случайные тензоры (fixed seed), каждый выход
   allclose со средним входных, atol=1e-6.
2. Число пересылок: счётчик on_send == 2*(N-1)*N для N=3 и N=5.
3. Не делится нацело: тензор длины 10 при N=4 — корректный результат.
4. Иммутабельность: входные тензоры после вызова не изменились
   (сравнить с клонами).
5. N=1: выход равен входу, on_send не вызывался.

## Подсказки (3 уровня)
1. «Почему наивный all-to-all шлёт O(N²) данных, а кольцо — O(N)?»
2. Схема двух фаз для N=3 (таблица: итерация × ранк × какой чанк едет).
3. Индексная арифметика: на итерации t ранк i отправляет чанк
   (i - t) mod N (reduce-scatter) — без готового кода.
