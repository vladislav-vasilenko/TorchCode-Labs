# 000: Конвенции проекта torchcode-labs (читать перед КАЖДОЙ сессией)

## Что это за проект

Форк https://github.com/duoan/TorchCode → репозиторий `torchcode-labs`.
Это ЕДИНСТВЕННАЯ рабочая директория: весь код, тесты и ноутбуки пишутся
только сюда.

Учебная платформа практики PyTorch в стиле LeetCode: template-ноутбук с TODO →
автопроверка judge → многоуровневые подсказки → эталонное решение.
Поверх 42 задач апстрима добавляются треки: `distributed`, `inference`,
`nanogpt`, `positional`.

Пользователь проекта — один человек, готовится к интервью Senior NLP
Researcher. Он решает задачи САМ. Твоя работа — инфраструктура, теория,
тесты judge, эталоны. НЕ решать за него интеграционные лабы (см. Анти-цели).

## Внешний репозиторий-цель (только чтение, НЕ рабочая директория)

Треки готовят пользователя к самостоятельному заполнению его лаб в
отдельном репозитории:
https://github.com/vladislav-vasilenko/modern_nlp_labs

Он нужен как контекст двум спекам:
- spec 108 (walkthrough) читает оттуда `distributed-train-lab/skeleton/
  train_my.py` (структура TODO 1–6) и `distributed-train-lab/STUDY.md`;
- spec 109 (тулинг результатов) читает формат
  `distributed-train-lab/results/*.json`.

Правила работы с ним: клонировать рядом с torchcode-labs только для
чтения; не коммитить в него, не предлагать туда PR, не копировать оттуда
эталонную реализацию `src/distributed_train_lab/train.py` ни целиком, ни
блоками.

## Рабочий процесс

- Один spec = одна ветка = один PR. Имя ветки: `spec/NNN-short-name`.
- В описании PR: какой spec реализован, как проверить (команды).
- Перед началом: прочитать этот файл + указанный spec + файлы, от которых
  spec зависит (секция «Зависит от»).
- Спеки выполняются строго по возрастанию номеров, если не сказано иначе.

## Формат задачи judge (как в апстриме)

Файл `torch_judge/tasks/<task_id>.py`:

```python
TASK = {
    "title": "Implement X",
    "difficulty": "Easy" | "Medium" | "Hard",
    "function_name": "fn_name",
    "track": "distributed",            # новое поле, см. spec 030
    "hints": ["level 1", "level 2", "level 3"],  # новое поле, см. spec 010
    "hint": "...",                     # legacy, оставить = hints[0]
    "tests": [
        {"name": "Test name", "code": "python-код с {fn} и assert'ами"},
    ],
}
```

К каждой задаче:
- `templates/NN_<task_id>.ipynb` — условие (русский), теор. вставка,
  пустая функция с `# TODO`, ячейка запуска judge, collapsible-подсказки
  (`<details><summary>Tip N</summary>...</details>`).
- `solutions/NN_<task_id>_solution.ipynb` — эталон с комментариями «почему
  так», а не только «что».

Нумерация NN шаблонов: 5X — distributed, 6X — inference, 7X — nanogpt и
positional. Не переиспользовать номера апстрима (01–4X).

## Жёсткие технические требования

1. **CPU-only.** Judge-тесты обязаны проходить на машине без GPU (CI и
   ноутбук пользователя — mac). Всё, что требует GPU, живёт в отдельных
   bench-скриптах, которые judge не запускает.
2. **Апстрим не ломать.** `pytest -q` всего репозитория зелёный после
   каждого PR; существующие 42 задачи и их шаблоны не изменяются.
3. **engine.py не переписывать.** Только расширения с обратной
   совместимостью (спеки 010, 030 описывают допустимые).
4. **Детерминизм тестов.** Фиксированные seed'ы; допуски allclose явные
   (atol указывать); никаких сетевых вызовов в judge-тестах (LLM-judge —
   исключение, и он мокается, см. spec 020).
5. **Тесты задач** покрывают минимум: формы, значения против эталона,
   градиенты (где применимо), один edge case.

## Языки и стиль

- Теория, условия задач, подсказки — русский.
- Код, идентификаторы, docstrings, сообщения assert — английский.
- Формулы — LaTeX в markdown-ячейках. Схемы — mermaid или ASCII.
- Теор. ноутбуки заканчиваются секцией «Контрольные вопросы» БЕЗ ответов
  в том же ноутбуке (ответы — в solution-ноутбуке трека, если уместно).

## Definition of Done (для каждого spec, если spec не сужает)

- [ ] `pytest -q` весь репозиторий — зелёный.
- [ ] Новые тесты spec'а — зелёные: `pytest tests/... -q`.
- [ ] Каждый новый template исполняется: `jupyter nbconvert --to notebook
      --execute templates/NN_*.ipynb` — без ошибок (TODO-функции при этом
      должны позволять исполнение ноутбука до ячейки judge: заглушка
      `raise NotImplementedError` допустима, ячейка judge в template
      обёрнута try/except с дружелюбным сообщением).
- [ ] Judge на эталонном решении: all pass (команда указывается в PR).
- [ ] README трека обновлён (порядок прохождения, что нового).

## Анти-цели (нарушение = провал spec'а)

- НЕ писать код в репозиторий modern_nlp_labs и не предлагать туда PR —
  он внешний и read-only (см. секцию выше).
- НЕ приводить готовые решения TODO 1–6 его скелета ни в каком файле
  torchcode-labs. В walkthrough (spec 108) допустимы: объяснения
  контракта, имена API, ловушки, иллюстративные фрагменты ≤ 2 строк.
  Недопустим: цельный работоспособный код любого TODO.
- НЕ генерировать текст для секций Results / Conclusions лаб пользователя
  ни в каком виде (включая «примеры хорошего ответа») — это его прогоны
  и его выводы; проверка выводов делается только рубриками LLM-judge.
- НЕ добавлять GPU-зависимости в judge.
- НЕ менять публичные сигнатуры/поведение апстрима.
- НЕ добавлять тяжёлые зависимости без необходимости (согласовывать в PR;
  разрешены: matplotlib, pyyaml, pytest, стандартный стек апстрима).
